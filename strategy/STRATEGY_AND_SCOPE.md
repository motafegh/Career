# Strategy and Scope

## Companion decision inputs

[`LEARNING_AND_PROJECT_DESIGN_PROFILE.md`](LEARNING_AND_PROJECT_DESIGN_PROFILE.md) records evidence about Ali's mission-driven learning mode, motivation conditions, ownership risks, and project-fit requirements.

[`PROJECT_SELECTION_AND_CAPABILITY_SPECIFICATION.md`](PROJECT_SELECTION_AND_CAPABILITY_SPECIFICATION.md) defines the mandatory problem, mission, novelty, product, capability, skill, evaluation, ownership, scope, rejection, and scoring requirements for selecting a primary project.

Both files are required input for any formal project-selection or route review. Neither file independently changes the fixed identity, current project allocation, master plan, or weekly execution order. Any resulting change must follow the Execution Contract's change criteria and state the evidence, affected gate, smallest necessary modification, and consequences.

## 1. Fixed 90-day identity

> **AI-augmented Python/data/ML engineer developing secure engineering capability.**

This identity is intentionally narrower than the long-term ambition.

### Primary capability

Build reliable Python systems that transform raw data into validated, queryable, tested, and explainable outputs.

### Specialization

Use sound data and machine-learning methods when they add measurable value over explicit baselines.

### Application domain

Apply those capabilities to security evidence, detection, and controlled laboratory scenarios.

### Working method

Use AI heavily for teaching, critique, implementation assistance, and debugging while preserving human understanding, verification, direction, and responsibility.

## 2. Long-term direction

The long-term direction remains broad and hybrid:

- software engineering;
- data and ML engineering;
- AI implementation;
- secure engineering;
- security automation;
- system reasoning;
- eventual technical leadership.

These are not simultaneous 90-day tracks. The program builds the narrow foundation required to earn later breadth.

## 7. Evidence standard

By Day 90, the portfolio must demonstrate a bounded set of capabilities rather than broad technology exposure.

Required evidence:

- runnable Python package or clearly structured application;
- raw-to-normalized data path;
- validation and quarantine;
- SQLite persistence and useful SQL queries;
- rule-based detector;
- unit and integration tests;
- Docker scenario transfer;
- CI;
- reproducible demo;
- deliberate failure diagnosis;
- simple ML comparison only if justified;
- honest README and limitations;
- clear disclosure of AI assistance;
- user explanation and modification of important components.

## 8. Decision rule

A proposed activity belongs in the 90-day program only when it directly improves the active weekly gate or one of the capability priorities above.

If it is merely useful, interesting, prestigious, or potentially relevant later, defer it.

During a formal project-selection or route review:

1. use the Learning and Project Design Profile to test mission gravity, real-input fit, causal visibility, sustained engagement, and ownership feasibility;
2. use the Project Selection and Capability Specification to apply hard rejection gates, capability requirements, novelty standards, feasibility checks, and weighted comparison;
3. do not protect a technically coherent direction when repeated execution evidence shows detached learning or weak product pursuit;
4. do not select a motivating idea when it fails the real-problem, evaluation, safety, scope, or ownership gates.
