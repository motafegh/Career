# Program Monthly Plan

The 90 days use three exact 30-day program months rather than calendar months.

## Month 1 — Close foundations and establish the Python path

**Dates:** 2026-07-20 to 2026-08-18  
**Days:** 1–30  
**Minimum focused time:** 63 hours across the included production days, subject to recorded Red-mode exceptions

### Required outcomes

1. Close the namespace SSH checkpoint.
2. Stop further namespace expansion.
3. Create the Python application/package structure.
4. Preserve representative raw SSH fixtures.
5. Define the normalized authentication-event model.
6. Implement parsing, validation, and quarantine behavior.
7. Establish unit tests.
8. Begin SQLite schema work before the Day 30 review.

### Day 30 review — 2026-08-18

Judge:

- Was the namespace gate actually closed?
- Did Python implementation begin on schedule?
- Can the user explain and modify the parser?
- Are invalid records handled explicitly?
- Did planning/documentation remain subordinate to implementation?
- Were 15-hour weeks normally achieved?

### Month 1 failure response

If parser/validation is incomplete, cut optional documentation and carry only the blocking technical item into Month 2. Do not reopen namespace scope or create another project.

## Month 2 — Complete the vertical slice and transfer it to Docker

**Dates:** 2026-08-19 to 2026-09-17  
**Days:** 31–60  
**Minimum focused time:** 66 hours across the included production days, subject to recorded Red-mode exceptions

### Required outcomes

1. Finish SQLite persistence and foundational SQL.
2. Add run identity and evidence metadata.
3. Implement the time-window rule detector.
4. Produce a finding and concise report.
5. Complete raw-to-report integration tests.
6. Add replay behavior.
7. Reproduce the scenario in Docker.
8. Generate multiple runs with changed counts and timing.
9. Add CI.
10. Begin reliability closure.

### Day 60 review — 2026-09-17

Judge:

- Does one clean command execute the complete vertical slice?
- Can the user query and explain SQLite data?
- Are normal, invalid, and failed paths tested?
- Does Docker preserve scenario meaning?
- Is the project still bounded?
- Are AI assistance and personal ownership accurately separated?

### Month 2 failure response

If the vertical slice has not passed, ML is automatically removed from the active schedule until it does. Portfolio and reliability work remain; no advanced feature compensates for an incomplete pipeline.

## Month 3 — Reliability, honest ML, portfolio closure, and market calibration

**Dates:** 2026-09-18 to 2026-10-17  
**Days:** 61–90  
**Minimum focused time:** 66 hours across the included production days, subject to recorded Red-mode exceptions

### Required outcomes

1. Complete reliability and clean-state testing.
2. Build an explicit run-level dataset.
3. Establish the rule baseline.
4. Control leakage by run or scenario group.
5. Train only a justified simple model.
6. Make an honest ML-use decision.
7. Close the portfolio presentation.
8. Obtain external repository/setup feedback.
9. Submit 5–10 calibration applications.
10. Complete the Day 90 evaluation.

### Day 90 review — 2026-10-17

Judge:

- What can the user explain, modify, test, query, and diagnose?
- What remains dependent on AI?
- Is AegisLab finished at the declared boundary?
- Are public claims reproducible and conservative?
- What did external reviewers and applications reveal?
- Which next 90-day direction follows from evidence?

### Month 3 failure response

No new project begins. The next period starts by closing the smallest missing evidence gate, unless the Day 90 evaluation explicitly chooses another evidence-backed route.
