# Agent Instructions — Career

## Purpose

Operate this repository as Ali Rajabi's durable career-control and formal-review repository.

Do not operate UpgradePilot from Career.

## Instruction precedence

When instructions conflict, use:

1. safety, legal, privacy, credential, financial, health, cost, and platform constraints;
2. Ali's explicit current instruction;
3. the stable Career Execution Contract;
4. Career strategy and formal capability model;
5. the current explicit Career-review scope;
6. other Career records;
7. AI suggestions.

## Career responsibilities

Career owns:

- working career identity and role direction;
- workload/capacity commitments and formal review dates;
- primary flagship allocation;
- advanced-system career commitments;
- formal capability assessment and portfolio claim limits;
- coarse reviewed project/career state.

Career does not own:

- UpgradePilot's current implementation state;
- project milestones, detailed plans, or exact next actions;
- project sessions, blockers, or working memory;
- project specifications, ADRs, source, tests, or evidence logs;
- ordinary learning and technical decisions.

## Career review rule

Do not update Career because one UpgradePilot event occurred.

Ali must explicitly request a Career review or a durable career/program change.

During a Career review:

1. define the career-level question;
2. inspect only the UpgradePilot evidence necessary to answer it;
3. distinguish project progress from demonstrated capability;
4. record a coarse conclusion, evidence, assistance, limits, and next review trigger;
5. do not copy project continuation or detailed technical state into Career.

## Truth routing

- Career identity, workload, strategy, and review commitments: Career controls.
- Coarse state and formal capability conclusions: Career review record.
- UpgradePilot current continuation: UpgradePilot `MEMORY.md`.
- UpgradePilot actual behavior: current source, tests, commands, outputs, and environment.
- UpgradePilot requirements and decisions: project-local charter, plan, specifications, and ADRs.

Do not combine these into one long authority hierarchy.

## Capability assessment

Use the formal capability model only during an explicit review or when Ali asks for an assessment.

- Record a specific responsibility, not a broad topic.
- Do not infer ownership from exposure, immediate repetition, successful commands, AI-generated code, or passing AI-generated tests.
- Use the lean record by default.
- Use the extended ownership vector only for central capabilities, D3+ claims, disputed assessments, mixed ownership, or portfolio claims.
- Preserve transfer limits and assistance.

## Historical documents

Pre-selection profiles and selection specifications are historical decision inputs. They do not reopen project selection or control UpgradePilot operation.

AegisLab and Sentinel are historical evidence sources unless a new explicit Career decision assigns a bounded role.

## Document discipline

Before editing a Career file, ask:

1. Is this a career/program responsibility?
2. Has Ali requested a Career review or durable Career change?
3. Is the information coarse and stable enough to belong here?
4. Would the same fact change after an ordinary project test or session?

If the answer to the last question is yes, it normally belongs in UpgradePilot instead.

## Security and claims

- Keep Career records public-safe.
- Do not expose sensitive health, medication, financial, credential, or private routine details.
- Do not inflate capability, ownership, role readiness, or production claims.
- Preserve uncertainty and limitations.
- Do not execute untrusted repository code merely to assess a project.
